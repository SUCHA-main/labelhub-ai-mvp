# LabelHub AI MVP 项目总结

## 项目背景

数据标注是 AI 应用落地的关键环节。真实的数据标注流程通常包括任务创建、人工标注、质量检查、审核流转和统计分析。完整的商业标注平台（如 Labelbox、Scale AI）功能复杂，不适合短期 MVP 验证。

本项目用 MVP 的方式抽取核心流程，聚焦"AI 如何辅助人工审核"这个场景，实现一个可运行、可演示的原型。

## 需求分析

### 核心需求（MVP 阶段必须做）

| 需求 | 说明 |
|------|------|
| 任务创建 | 管理员定义任务标题、待标注文本和表单 schema |
| 动态表单标注 | 标注员根据 schema 自动渲染的表单填写标注结果 |
| AI 预审 | 标注提交后触发 AI 检查，返回结构化风险评估 |
| 人工审核 | 审核员查看 AI 建议，做出通过或驳回决定 |
| 统计展示 | Dashboard 展示任务状态分布和关键指标 |

### 非核心需求（MVP 阶段暂不做）

| 需求 | 说明 | 原因 |
|------|------|------|
| 真实认证 | 用户名密码、JWT、OAuth | MVP 重点是流程闭环，demo 角色登录足够 |
| 真实 AI | 大模型接入、规则引擎 | 先验证产品流程，mock 实现可以展示接口设计 |
| 数据库 | SQLite、PostgreSQL | JSON 文件降低部署复杂度 |
| 权限控制 | 角色权限、API 鉴权 | 前端角色切换已足够展示差异 |
| 并发控制 | 任务锁定、多人协作 | MVP 阶段单人演示足够 |
| 审计日志 | 操作历史、变更记录 | 后续扩展方向 |

## 产品流程

```
管理员创建任务（标题 + 文本 + schema）
         ↓
    任务进入"待标注"状态
         ↓
标注员查看待标注任务，填写动态表单
         ↓
    提交标注结果
         ↓
系统自动触发 mock AI 预审
         ↓
    任务进入"待审核"状态
         ↓
审核员查看原文、标注结果和 AI 预审卡片
         ↓
    做出"通过"或"驳回"决定
         ↓
    Dashboard 统计更新
```

## 技术架构

| 层级 | 技术 | 职责 |
|------|------|------|
| 前端 | React 18 + Vite + Tailwind CSS | 角色工作台、动态表单渲染、Dashboard 展示 |
| 后端 | Node.js + Express | API 接口、业务逻辑、mock AI 调用 |
| 数据存储 | JSON 文件 | 任务数据持久化（MVP 方案） |
| AI 预审 | 规则函数 | 关键词检测 + 长度判断，返回结构化结果 |
| 部署 | Docker Compose + Nginx | 前后端容器编排、反向代理 |

## 核心功能

### 1. 三角色工作流

- **Admin（管理员）**：创建任务、查看 Dashboard、查看全部任务
- **Annotator（标注员）**：查看待标注/被驳回任务、填写动态表单、提交标注
- **Reviewer（审核员）**：查看待审核任务、查看 AI 预审结果、通过或驳回

三个角色各有独立工作台，职责清晰分离。

### 2. 动态表单

任务通过 JSON schema 定义表单结构，支持 5 种字段类型：

| 类型 | 用途 |
|------|------|
| text | 短文本输入 |
| textarea | 长文本输入 |
| select | 下拉选择 |
| number | 数字输入 |
| checkbox | 布尔选择 |

新增标注任务类型只需要定义新的 schema，不需要改前端代码。

### 3. Mock AI 预审

标注提交后自动触发 `mockAIReview` 函数，返回结构化结果：

```javascript
{
  riskLevel: '低' | '中' | '高',
  confidence: 0.88,
  possibleIssue: '标注内容完整，暂未发现明显一致性风险。',
  suggestion: '建议进入人工审核确认。'
}
```

### 4. Dashboard 统计

展示 8 个指标：任务总数、待标注、待审核、已通过、已驳回、通过率、AI 风险提示数、待处理任务数。

## 核心实现说明

### 6.1 动态 Schema 表单

**设计思路**：任务对象中包含 `schema` 字段，是一个 JSON 数组，定义每个表单字段的名称、标签、类型、选项和是否必填。前端 `DynamicField` 组件根据 `field.type` 渲染对应的 HTML 元素。

**好处**：
- 不同标注场景（情感分类、内容安全、标题质量）共用同一套前端代码
- 管理员可以灵活定义表单结构
- 前端组件逻辑集中在 `DynamicField`，易于维护

### 6.2 Mock AI 审核

**设计思路**：`mockAIReview` 函数接收标注结果对象，通过关键词检测和字符串长度判断风险等级，返回 4 个维度的结构化结果。

**实现逻辑**：
1. 拼接所有标注值为字符串
2. 检查是否包含风险关键词（"不确定"、"不知道"、"违规"等）→ 高风险
3. 检查是否有答案长度 < 2 字符 → 中风险
4. 否则 → 低风险
5. 根据风险等级返回对应的置信度、问题提示和修改建议

**为什么不直接接真实 AI**：
- 避免依赖外部 API Key
- 降低演示不稳定性
- 先验证产品流程，再替换 AI 实现
- 接口设计已经预留了替换空间

### 6.3 三角色流程

**设计思路**：前端通过 `session.user.role` 判断当前角色，渲染对应的工作台组件。后端 API 按角色提供不同视角的数据（如 annotator 只看 pending_annotation 和 rejected，reviewer 只看 pending_review）。

**好处**：
- 职责分离清晰
- 演示时可以展示不同角色的视角差异
- 贴近真实标注团队的协作模式

### 6.4 Dashboard 统计

**设计思路**：`/api/dashboard` 接口读取全部任务，按状态分组计算各指标。前端用卡片网格展示，每张卡片有 emoji 图标和彩色背景。

**指标说明**：
- 通过率 = 已通过 / (已通过 + 已驳回)，体现标注质量
- AI 风险提示数 = 中 + 高风险任务数，体现 AI 发现问题的能力
- 待处理数 = 待标注 + 待审核 + 已驳回，体现工作负载

## 技术取舍

### 7.1 为什么用 JSON 文件

**原因**：
- 降低部署复杂度，`docker compose up` 就能跑
- 数据访问集中在 `dataStore.js` 的 `readTasks` / `writeTasks`
- MVP 阶段不需要数据库的事务、索引、备份能力
- 后续替换为数据库只需要改 dataStore 层

**代价**：
- 无并发控制
- 无数据备份
- 文件读写性能有限

### 7.2 为什么用 Mock AI

**原因**：
- 避免依赖外部 API Key，降低演示门槛
- 避免 API 调用延迟和失败风险
- 先验证"AI 预审层在产品流程中的位置"这个假设
- 接口输出结构（riskLevel / confidence / possibleIssue / suggestion）已经设计好，后续替换只需要改函数实现

**代价**：
- AI 预审结果是基于规则的，不够智能
- 无法展示真实大模型的能力

### 7.3 为什么不做完整认证

**原因**：
- MVP 阶段重点是流程闭环，不是用户管理
- Demo 角色登录已经足够展示三个角色的差异
- 真实认证需要密码加密、token 管理、权限校验，增加复杂度

**代价**：
- 无真实权限控制
- 任何人可以调用任何 API

### 7.4 为什么先做前后端分离

**原因**：
- 更接近真实项目结构
- 前端 React + 后端 Express 职责清晰
- 方便后续独立扩展（前端换框架、后端换语言）
- Vite 代理机制简单易懂

**代价**：
- 需要同时启动两个服务
- 跨域处理（已通过 cors 中间件解决）

## 遇到的问题与解决思路

### 问题 1：如何让不同标注任务共用同一套表单

**解决**：使用 JSON schema 定义表单结构，前端 `DynamicField` 组件根据 `field.type` 渲染不同控件。新增任务类型只需要定义 schema，不需要改前端代码。

### 问题 2：如何展示 AI 价值但不依赖真实 AI

**解决**：设计 `mockAIReview` 函数，用规则模拟 AI 输出。重点是保留"AI 预审层"的接口位置和输出结构，让评委看到 AI 在产品流程中的作用点。

### 问题 3：如何让演示流程清楚

**解决**：三角色拆分，每个角色一个独立工作台。演示时按 Admin → Annotator → Reviewer 顺序切换，展示完整闭环。

### 问题 4：如何避免项目太重

**解决**：JSON 文件存储 + Docker 本地演示 + 无认证系统。保持项目轻量，聚焦核心流程。

## 当前局限

诚实列出当前项目的边界：

- **Mock AI**：基于规则的关键词检测，不是真实大模型
- **JSON 文件存储**：不是生产数据库，无事务、无备份
- **Demo 登录**：不是真实认证系统，无密码、无 token 验证
- **无权限控制**：前端按角色隐藏面板，但后端 API 无鉴权
- **无并发控制**：无任务锁定，多人同时操作可能冲突
- **无审计日志**：无操作历史记录
- **无自动化测试**：测试是后续改进方向
